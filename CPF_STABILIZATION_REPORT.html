CPF 안정화 작업 리포트

검수 결과 기반 보정 요청에 따라 구현 상태와 검증 상태를 분리하고, SQL 설치 정합성, EDU Mapper 검증 구조, 표준 헤더 보안 정책, 기능 매트릭스를 보강했습니다.

실행하지 않은 ADM runtime, OpenAPI runtime, 브라우저 클릭, MariaDB 신규 DB 설치, Redis/Kafka/MQ 실 broker 검증은 성공으로 기록하지 않았습니다.

수행 작업

영역반영 내용상태
리포트/매트릭스specs/기능_구현_매트릭스.html를 실제 구현과 검증 상태 기준으로 재정리했습니다. 정적 smoke, runtime smoke, 브라우저 클릭, MariaDB 실설치를 별도 행으로 분리했습니다.반영
SQL 정합성cmn_edu_query_item이 split SQL, Flyway, all install, all install and smoke에 모두 있는지 재확인했습니다. check-sql-standard.ps1에 split schema 테이블이 전체 설치본과 baseline에 누락되면 실패하는 검사를 추가했습니다.반영
EDU MapperXyzQueryEducationMapperSliceTest와 xyz_edu_query_mapper_fixture.sql을 추가했습니다. XML의 안전한 정렬 whitelist는 기본 테스트에서 검증하고, 실제 DB fixture 실행은 안전한 테스트 DB 환경변수 설정 시에만 수행합니다.반영
표준 헤더CpfTrustedProxyPolicy를 추가해 trusted proxy가 아닌 요청에서는 전달 IP 헤더를 신뢰하지 않도록 했습니다. LoggingAspect와 CpfHeaderExtractor도 같은 정책을 사용합니다.반영
헤더 보정 제한CpfHeaderMutator에서 X-User-Id, X-Operator-Id, correlation, idempotency 등 감사/추적 주체성 헤더의 업무 코드 변경을 차단했습니다.반영
민감 헤더inbound header 추출 단계에서도 토큰/API key/서명/nonce류 원문이 저장 경로에 올라가지 않도록 마스킹을 적용하고 테스트를 추가했습니다.반영
문서표준 헤더 가이드, 개발 가이드, SQL 가이드, 기능 구현 매트릭스를 구현 내용에 맞춰 갱신했습니다. README는 짧은 진입점 원칙을 유지했습니다.반영

미검증/보류

항목상태사유/다음 조치
MariaDB 신규 DB 전체 설치미검증all install and smoke는 cleanup 위험이 있으므로 안전한 빈 DB 또는 명시 승인 후 실행해야 합니다.
EDU Mapper 실제 DB slice미검증안전한 테스트 DB 환경변수가 없어 실제 fixture 실행은 skip되었습니다.
ADM runtime smoke미검증ADM 앱 기동과 API 호출 검증은 이번 작업에서 아직 실행하지 않았습니다.
OpenAPI runtime smoke미검증앱 기동 후 OpenAPI JSON 접근 검증은 아직 실행하지 않았습니다.
ADM 브라우저 클릭미검증정적 marker 검사는 브라우저 클릭 검증으로 보지 않습니다.
로그 정책 runtime smoke미검증로그 정책 override, fallback, ADM 조회 runtime 검증은 아직 실행하지 않았습니다.
Redis/Kafka/MQ 실 broker실환경 필요실 broker와 장애 시나리오가 필요합니다.

다음 보강 후보

순위항목이유
1ADM runtime/OpenAPI/브라우저 클릭 자동화운영자 기능은 정적 marker가 아니라 실제 API와 화면 검증이 필요합니다.
2안전한 MariaDB 테스트 DB에서 전체 설치, seed idempotent, 권한 분리 검증신규 프로젝트 설치 재현성을 확정해야 합니다.
3표준 헤더 E2E 전파 검증수신, context, 로그 detail, ADM 응답, outbound 전파를 한 거래에서 증명해야 합니다.
4EDU Mapper DB slice 실실행 자동화개발자 교육 샘플이 XML/fixture 기반으로 완전히 검증되어야 합니다.
5배치 ghost scheduler, 장애 시나리오, 관계구조도 UX운영 관제 완성도를 높이는 다음 핵심입니다.
6broker 기반 로그/캐시 정책 전파다중 인스턴스 운영에서는 DB fallback 외 실시간 전파 검증이 필요합니다.
