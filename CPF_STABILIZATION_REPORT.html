CPF 안정화 작업 리포트

작성일: 2026-06-19

CPF_REQUEST.md는 요청 확인용으로만 사용했고 작업 대상에서 제외했습니다. 별도 수정파일 목록 파일은 생성하지 않았습니다.

1. 작업 요약

구분내용
[성공]BIZADM/MBR 인증 기본 API, EXS 운영 API, 인증/token SQL 스키마, Flyway V7, 합본 설치 SQL, smoke 조회, 품질 gate, HTML 가이드를 동기화했습니다.
[일부 구현]신규 API는 공통 JWT/암호화/거래 ID 기준을 사용하지만 token 저장소는 현재 in-memory 기본 구현입니다. 실제 DB mapper 영속화와 Vue 화면 연결은 다음 단계입니다.
[미검증]MariaDB 실실행, 앱 기동 후 OpenAPI JSON 검증, 브라우저 클릭 검증, Redis/Kafka/외부기관 실연동은 이번 환경에서 수행하지 않았습니다.

4. 실패 항목

  • Node 기반 JS 문법 검사 실패: node 명령이 현재 PATH에 없어 node --check adm/src/main/resources/static/adm/adm.js를 실행하지 못했습니다.
  • 일시 이슈: 이전 병렬 Gradle 검증은 타임아웃 가능성이 있어 중단했고, 동일 검증은 순차 실행으로 재수행해 성공했습니다.

5. 미검증 항목

항목미검증 사유다음 조치

OpenAPI JSON 품질 gate앱 기동과 /v3/api-docs 호출을 수행하지 않았습니다.ADM/BIZADM/MBR/EXS 앱 기동 smoke와 JSON schema 검사를 자동화합니다.
Vue/브라우저 클릭 검증브라우저 자동화와 Node 도구가 준비되지 않았습니다.Node/Playwright 또는 프로젝트 표준 UI smoke 환경을 구성합니다.
Redis/Kafka/외부기관 연동실 broker와 외부기관 mock server를 실행하지 않았습니다.Docker compose 기반 broker/mock smoke를 분리 추가합니다.

7. 설계 판단 내역

판단근거
BIZADM/MBR token 분리ADM, BIZADM, MBR token 혼용을 막기 위해 loginDomain, issuer, audience를 분리했습니다.
refresh token hash 저장원문 refresh token 저장을 피하고, 응답 시 최초 1회만 원문을 반환하는 구조로 잡았습니다.
EXS token과 사람 로그인 token 분리대외기관 token은 운영 API에서 hash/마스킹 중심으로 다루고 사람 인증 token과 혼용하지 않습니다.
합본 SQL 생성 스크립트00_all_install.sql, 00_all_install_and_smoke.sql, Flyway baseline의 drift를 줄이기 위해 split SQL 기반 재생성 스크립트를 추가했습니다.
생성 스크립트 주석 일부 ASCII 유지Windows PowerShell 5.1이 BOM 없는 PS1의 한글 here-string을 ANSI로 해석해 합본 SQL에 mojibake가 생기는 문제를 막기 위한 예방 조치입니다.

8. 재확인 필요 항목

  • 현재 SQL 서비스 계정 예시는 local 설치용 고정 비밀번호를 사용합니다. 운영 배포에서는 반드시 환경변수/Vault/KMS 또는 배포 시 치환 구조로 분리해야 합니다.
  • BIZADM/MBR 인증 서비스는 in-memory 저장소이므로 다중 WAS와 재기동 환경에서는 DB/Redis 저장소로 교체해야 합니다.
  • 기능 매트릭스의 BIZADM/EXS UI 항목은 일부 구현으로 유지했습니다. API가 생겼지만 화면 route와 실제 클릭 smoke는 아직 아닙니다.

11. Vue 화면 반영 내역

[일부 구현/미구현] 이번 작업은 API, SQL, Swagger annotation, 문서 중심입니다. BIZADM/EXS 전용 Vue route와 화면 클릭 검증은 수행하지 않았습니다. ADM 기존 화면은 이번 범위에서 수정하지 않았습니다.

12. BIZADM 구현 수준

[일부 구현] 로그인, refresh, logout, 현재 사용자, 로그인 이력 API와 SQL 저장 구조를 추가했습니다. 운영자/역할/메뉴/권한/다운로드/마스킹/감사 UI의 완성 구현은 다음 단계입니다.

13. MBR 구현 수준

[일부 구현] 회원 로그인과 token 재발급, 현재 회원, 로그인 이력 API를 추가했습니다. 회원 상태별 로그인 제한, DB mapper 기반 token 영속화, 회원 다운로드/마스킹/감사 UI는 다음 단계입니다.

14. EXS 구현 수준

[일부 구현] 대외 token 운영, token 이벤트, 통제 정책, 재처리 API와 저장 구조를 추가했습니다. 실제 기관 adapter, 라우팅 엔진, broker 연동, 전문 파싱/재처리 UI는 다음 단계입니다.

15. 공통 인증/토큰 구현 수준

[일부 구현] 공통 JWT/crypto 서비스를 재사용하고 token에 loginDomain, moduleId, wasId, serverInstanceId를 포함했습니다. 공통 refresh token repository 추상화와 서버 권한 필터 통합은 추가 보강이 필요합니다.

16. 공통 코드/메시지 구현 수준

이번 작업에서 공통 코드/메시지 관리 기능은 직접 변경하지 않았습니다. 기존 PFW/ADM 기준을 유지했고, 신규 오류/응답 메시지의 DB seed화는 다음 보강 과제로 남겼습니다.

17. transactionGlobalId 구현 수준

신규 BIZADM/MBR 로그인 이력과 EXS 운영 이벤트에 transactionGlobalId를 포함했습니다. 서비스 코드는 현재 거래 ID가 없으면 TransactionContext.getOrCreateTransactionId 기준으로 생성합니다.

18. 서버 인스턴스 추적 구현 수준

신규 API 응답/이력 모델에 moduleId, wasId, serverInstanceId를 포함했습니다. 이중화 운영에서 어느 인스턴스가 token/로그를 처리했는지 추적하는 기준을 마련했습니다.

20. 기능 구현 매트릭스 반영 수준

기능 구현 매트릭스에 MBR 인증, BIZADM 인증, EXS 운영 API, 보안 seed gate를 반영했습니다. 구현 수준은 실제 상태에 맞춰 일부 구현과 미검증으로 표시했습니다.

21. 잔여 과제

  1. MariaDB client 경로 확인 후 00_all_install_and_smoke.sql 실실행, FK/index/seed idempotent/권한 분리 검증
  2. BIZADM/MBR refresh token 저장소를 DB mapper 또는 Redis 기반으로 교체하고 다중 WAS 재기동 시나리오 테스트
  3. BIZADM/EXS Vue 화면 route, 권한별 버튼 상태, 다운로드/마스킹/감사 UI 구현
  4. OpenAPI JSON 자동 검증 task와 앱 기동 smoke 추가
  5. Node/브라우저 UI 검증 환경 구성 후 ADM/BIZADM/EXS 화면 클릭 검증
  6. 운영 secret/JWT/DB password를 prod profile에서 기본값 없이 환경변수/Vault/KMS로 강제
  7. Redis/Kafka/외부기관 mock server를 Docker compose로 구성하고 fallback 장애 시나리오 검증
  8. 공통 오류코드/메시지 seed와 신규 API 응답 메시지 연결
