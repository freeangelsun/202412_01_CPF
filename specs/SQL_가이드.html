SQL 가이드

specs/sql은 로컬/검증용 MariaDB 설치 SQL과 운영 migration 기준을 함께 관리합니다. 분리 SQL은 유지보수 기준이고, 00_all_install.sql, 00_all_install_and_smoke.sql는 실제 SQL 본문을 모두 포함한 단일 실행 파일입니다.

테이블/컬럼 표준

항목기준
테이블명{주제영역}_{업무명} lower snake case를 사용하고 *_table suffix는 사용하지 않습니다.
제약조건FK, UK, index는 fk_, uk_, ix_ + 테이블명 + 의미 순서로 작성합니다.
공통 컬럼모든 테이블은 created_by, created_at, updated_by, updated_at을 가집니다.
한글 COMMENT모든 테이블과 주요 컬럼은 DB에서 바로 읽을 수 있도록 한글 COMMENT를 작성합니다.
Spring Batch 예외BATCH_*는 외부 프레임워크가 고정 명칭으로 찾기 때문에 대문자 표준명을 유지합니다.
거래 로그 일자pfw_transaction_log.LOG_DATE는 적재 시각이 아니라 START_TIME의 날짜와 같아야 합니다. smoke에서는 불일치 건수를 조회합니다.

CMN 채번 기준

cmn_sequence는 sequence_key 직접 호출과 business_area + business_key + sequence_kind + channel_code 조합 호출을 모두 지원합니다.

항목기준
시작/증가/범위start_value, increment_by, min_value, max_value로 번호 범위를 제어합니다.
반복주기reset_cycle은 NONE, DAY, MONTH, YEAR, PATTERN을 기준으로 사용합니다.
발급 로그log_enabled_yn='Y'이면 cmn_sequence_issue_log에 발급 이력을 남깁니다.
gap 정책발급번호는 gap을 허용합니다. 업무 처리 실패 후 번호 재사용은 감사 추적을 해치므로 금지합니다.

Flyway 기준

Flyway를 표준 migration 도구로 채택합니다. 운영 적용 시 migration 계정만 DDL 권한을 가지고, 애플리케이션 계정은 DML 권한만 가집니다.

  1. V5__add_xxx.sql처럼 증가하는 버전 파일을 추가합니다.
  2. 같은 변경을 분리 설치 SQL에도 반영합니다.
  3. 00_all_install.sql, 00_all_install_and_smoke.sql, Flyway baseline 또는 증분 migration을 함께 갱신합니다.
  4. MariaDB에서 idempotent, FK/index, seed, 권한을 확인합니다.
원칙기준
설치용 SQLspecs/sql/*.sql에서 관리하고 신규 설치 편의 파일에 본문을 모두 반영합니다.
운영 변경 이력specs/sql/migration/flyway에 V{버전}__{설명}.sql 형식으로 누적합니다.
rollback자동 역방향 SQL보다 백업/복구와 보정 migration을 우선합니다.
archivespecs/sql/archive는 과거 실험 DB나 초기 개발 단계의 레거시 객체 정리 참고 SQL만 보관하고, 공식 신규 설치 흐름에는 포함하지 않습니다.

Spring Batch 저장소와 CPF 운영 메타

구분기준
표준 저장소Spring Batch가 직접 사용하는 BATCH_JOB_INSTANCE, BATCH_JOB_EXECUTION, BATCH_STEP_EXECUTION 같은 BATCH_* 테이블입니다.
CPF 운영 메타ADM 관제에서 검색하고 보여주는 운영 정보는 pfw_batch_job, pfw_batch_schedule, pfw_batch_execution, pfw_batch_step_execution, pfw_batch_operation_log에 저장합니다.
실행 연결pfw_batch_execution.spring_batch_execution_id는 Spring Batch BATCH_JOB_EXECUTION.JOB_EXECUTION_ID를 연결하는 참조 값입니다.
영업일pfw_batch_schedule.business_day_only_yn='Y'이면 pfw_business_day_calendar 기준 영업일만 수행 후보로 계산합니다.
관계/대상pfw_batch_job_relation은 선행/후행/트리거 관계를, pfw_batch_execution_target은 수행 대기/배정/완료 대상 인스턴스를 저장합니다.

Smoke 실행 예시

운영에서는 비밀번호가 콘솔 기록에 남지 않도록 secret 주입 방식을 사용합니다.

cmd.exe /d /c '"C:\Program Files\MariaDB 12.3\bin\mariadb.exe" --host=127.0.0.1 --port=3306 --user=root --password=<비밀번호> --ssl=0 --default-character-set=utf8mb4 < "specs\sql\00_all_install_and_smoke.sql"'

캐시 이벤트 테이블

pfw_cache_refresh_event는 Redis/Kafka가 없거나 장애가 있는 환경에서도 다중 WAS 캐시 갱신을 추적하고 재처리하기 위한 DB fallback 이벤트입니다. broker가 있으면 broker를 우선 사용하고, DB event는 추적과 fallback 용도로 유지합니다.

이번 기준 스키마 보강

ADM 다운로드 감사adm_download_audit_log는 다운로드 요청자, 메뉴, 유형, 검색 조건, 행 수, 마스킹 여부, 민감정보 포함 요청 여부, 사유, IP, User-Agent, 상태, 실패 사유, 파일명을 기록합니다.
알림/다운로드 권한 SeedNOTIFICATION, DOWNLOAD 메뉴와 NOTIFICATION_*, DOWNLOAD_*, BATCH_SCHEDULER_RUN 버튼 권한을 seed에 추가했습니다.
Flyway신규 설치 기준은 baseline SQL에 반영하고, 기존 DB 반영 기준은 V5__adm_download_scheduler_quality_gate.sql, V6__bizadm_exs_transaction_identity.sql처럼 idempotent DDL/seed로 분리합니다.
단일 설치 SQL00_all_install.sql, 00_all_install_and_smoke.sql는 SOURCE 호출이 아닌 전체 SQL 포함 파일로 유지합니다.
