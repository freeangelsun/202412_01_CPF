CPF 개발 가이드

CPF는 cpf.* Java package를 표준으로 사용합니다. 기존 프레임워크 API 호환을 위해 FpsTransaction, FpsWebClient, FpsWorkflow 같은 클래스명은 유지합니다.

1. 개발 기준

항목기준
JavaJava 21
SpringSpring Boot 3.4.13
Root projectcpf-coreflow-platform-framework
공통 모듈pfw, cmn
업무/운영 모듈acc, mbr, xyz, adm
설정 prefix신규 설정은 cpf.*를 사용하고, 환경변수는 가능한 경우 CPF_* 우선, FPS_* fallback을 둡니다.

2. 패키지 규칙

패키지는 cpf.{module}.{domain}.{layer} 형태를 사용합니다.

예시용도
cpf.pfw.common.logging거래 로그, 거래 컨텍스트, 동적 로그레벨
cpf.cmn.msg.service메시지/응답코드 캐시와 resolver
cpf.mbr.bse.controller회원 기본 샘플 API
cpf.acc.tst.controllerACC 테스트/교육 API
cpf.xyz.edu.controller교육용 기능별 샘플 API
cpf.adm.opr.controller운영자 로그/캐시/응답코드 관리 API

3. 코드/메시지 표준

분류형식예시
응답코드{S|E}{MODULE}{GROUP}{SEQ}SACC000000, EACC010001, EPFW900001
메시지코드M{MODULE}{GROUP}{SEQ}MCMN000001, MXYZ090001
거래코드{MODULE}{TYPE}{DOMAIN}{SEQ}MBR01BSE0001, ADM05OPR0021

response_code_table은 응답코드와 메시지코드를 연결합니다. message_table은 한 row에 external_message, internal_message를 함께 저장하며 {0}, {1} indexed parameter를 지원합니다.

@FpsTransaction(id = "MBR01BSE0002", name = "MBRMemberDetail")
public BaseResponse<Member> detail(@RequestParam Long memberId) { ... }

4. 응답/헤더

PFW/CMN resolver는 응답코드/메시지코드를 캐시에서 해석해 응답 body, 응답 header, 거래 로그 metadata에 반영합니다.

헤더설명
X-Fps-Response-Code표준 응답코드. 성공/실패 모두 가능
X-Fps-Response-Message-Code응답코드에 연결된 메시지코드
X-Fps-Message-Code실제 출력 메시지코드
X-Fps-Message-Content외부 메시지 내용 URL-encoded 값

7. DB/SQL

개발자 로컬 설치는 아래 명령 하나로 시작할 수 있습니다.

mysql -h localhost -P 3306 -u root -p < specs/sql/00_all_install_and_smoke.sql

DBA 검토나 부분 재실행이 필요하면 01_create_databases.sql부터 70_test_data.sql까지 순서대로 실행하고 마지막에 99_smoke_check.sql을 실행합니다.

8. 검증

.\gradlew.bat :pfw:compileJava :cmn:compileJava :adm:compileJava :acc:compileJava :mbr:compileJava :xyz:compileJava --offline
.\gradlew.bat :mbr:test --tests cpf.cmn.sec.crypto.CmnCryptoServiceTest --tests cpf.cmn.msg.service.CmnFpsResponseCodeResolverTest --tests cpf.pfw.common.logging.LoggingAspectMetadataTest --tests cpf.mbr.bse.controller.MbrControllerValidationTest --offline
.\gradlew.bat :acc:test --offline
오프라인 빌드는 로컬 Gradle cache에 의존합니다. 의존성이 없으면 네트워크가 가능한 환경에서 한 번 의존성을 내려받아야 합니다.

인코딩과 IDE Problems

소스와 문서는 UTF-8을 기준으로 관리합니다. 루트 .editorconfig가 문자셋과 줄 끝을 고정하고, scripts/check-utf8.ps1이 UTF-8 디코딩 오류를 검사합니다.

.\scripts\check-utf8.ps1
.\scripts\check-utf8.ps1 -CheckMojibake

VS Code Java의 null analysis는 프로젝트 초기 보강 중 Problems 패널에 보수적인 경고를 많이 올릴 수 있어 로컬 .vscode/settings.json에서 java.compile.nullAnalysis.mode를 disabled로 둡니다. 실제 null 방어는 TextUtils.hasText, DTO validation, controller/service guard clause로 처리합니다.
