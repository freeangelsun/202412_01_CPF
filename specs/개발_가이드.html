CPF 개발 가이드

이 문서는 CPF 기반 신규 업무 개발자가 어떤 모듈, API, EDU 샘플을 참고해야 하는지 연결합니다. 개발자는 PFW 코어를 직접 수정하기보다 PFW가 제공하는 공통 API를 사용하고, 프로젝트 공통 확장은 CMN에 둡니다.

신규 업무 모듈 작성 기준

단계작성 대상기준참고 샘플
APIController, DTO업무 경로는 /api/{domain}, 운영 API는 /adm/api/{domain} 형식을 사용합니다. 모든 운영 API는 @CpfTransaction 메타를 둡니다.xyz/edu/controller/XyzCrudEducationController.java
Service업무 규칙, 트랜잭션트랜잭션 경계는 Service 계층에 둡니다. 외부 호출과 DB lock을 긴 트랜잭션으로 묶지 않습니다.xyz/edu/controller/XyzTransactionEducationController.java
Repository/MapperDB 접근업무 테이블은 업무 DB에 두고, PFW 테이블은 PFW API를 통해 사용합니다.mbr/src/main/java/cpf/mbr/bse/mapper
ValidationRequest DTO, validator입력값 검증은 Controller 진입 시점에 수행하고 오류 응답은 CPF 표준 예외로 통일합니다.XyzSecurityEducationController
SwaggerTag, summary, description운영자가 읽을 수 있는 한글 설명과 request/response example을 둡니다.AdmBatchController
Test단위, 슬라이스, 통합중요 서비스는 단위 테스트, API는 validation 테스트, DB 기능은 가능한 경우 통합 테스트를 둡니다.adm/src/test/java, cmn/src/test/java

온라인 거래 메타 개발 기준

항목현재 기준개발자가 지킬 점
거래 IDController 또는 메서드에 @CpfTransaction을 선언합니다.ID는 {모듈3자리}{업무구분2자리}{거래구분3자리}{일련번호4자리} 형식을 사용하고 이름은 운영자가 이해할 수 있게 씁니다.
거래 글로벌 IDTransactionContextFilter가 X-Transaction-Id를 검증하거나 생성하고 응답 헤더로 돌려줍니다.서비스 간 호출은 TransactionContext.propagationHeaders()를 사용해 trace와 거래 헤더를 이어갑니다.
필수 거래 헤더X-Transaction-Id, X-Request-Type, X-Original-Channel-Code, X-Channel-CodeTransactionHeaderValidationInterceptor가 누락과 거래 ID 형식을 검사합니다. ADM UI와 smoke는 호출 시 이 값을 기본으로 보냅니다.
권장 거래 헤더X-Trace-Id, X-Parent-Span-Id, X-Client-Version, X-Caller-Service, X-Correlation-Id, X-Idempotency-Key, X-Member-No, X-Customer-No, X-User-Id값이 있으면 TransactionContext와 propagationHeaders()로 이어지고, 로그 상세의 헤더부에서 확인할 수 있습니다.
ADM 로그 표시LoggingAspect가 headers, requestBody, response, error, logPolicy.* detail을 저장합니다.AdmLogQueryService는 상세 응답을 요약, 헤더, 요청, 응답, 오류, 상세, 전문 영역으로 나눠 반환합니다.
거래 로그LoggingAspect가 요청, 응답, 오류, 실행 메타를 수집하고 pfw_transaction_log에 저장합니다.민감정보가 들어갈 수 있는 값은 DTO 단계와 로그 저장 단계 모두에서 마스킹 기준을 확인합니다.
메타 자동 등록CpfTransactionMetaScanner가 Spring MVC RequestMapping과 @CpfTransaction을 스캔해 pfw_transaction_meta에 upsert합니다.신규 API를 만들 때 거래 ID, 거래명, Swagger 설명을 함께 작성해야 ADM 거래 메타에서 식별하기 쉽습니다.
로그 정책LogPolicyResolver가 ADM 활성 override, DB 운영 정책, application 기본값, CPF 기본값 순서로 ONLINE_TRANSACTION 정책을 평가합니다.dbLogEnabled=N이면 거래 로그 DB 저장을 생략하고, requestBodySaveYn/responseBodySaveYn/errorStackSaveYn 값에 따라 본문과 내부 오류 상세 저장 여부가 결정됩니다.

배치 개발 기준

항목기준구현 위치
실행 책임실제 Job/Step 실행은 BAT가 담당합니다. ADM은 조회, 수동 실행 요청, 재수행 요청, 중지 요청 같은 관제와 조치만 담당합니다.bat/src/main/java/cpf/bat, adm/src/main/java/cpf/adm/opr/controller/AdmBatchController.java
공통 API 사용BAT는 Spring Batch를 직접 호출하더라도 CPF 완료 기준에서는 CpfBatchLauncher를 통해 실행해야 합니다.BatSmokeOperationService
JobRepositorySpring Batch 원천 메타는 BATCH_*에 기록합니다.BatBatchRepositoryConfig
운영 메타CPF 관제 메타는 pfw_batch_execution, pfw_batch_step_execution에 기록합니다.CpfBatchOperationRepository
최소 검증정상 Job과 실패 Job을 모두 실행해 Spring Batch execution id, step execution id, CPF execution id, transactionGlobalId 연결을 확인합니다.scripts/smoke-bat-runtime.ps1
로그 정책 hookCpfBatchRuntimeListener가 Job 시작 시 BATCH_JOB, Step 시작 시 BATCH_STEP 정책을 조회해 ExecutionContext에 남깁니다.배치 로그 저장소 고도화 시 같은 key를 사용해 Job/Step별 저장 여부와 레벨을 적용합니다.

표준 헤더와 컨텍스트 사용

온라인 API 개발 시 헤더 문자열을 직접 만들지 말고 CpfHeaderNames, TransactionContext, CpfHeaderMutator를 사용합니다. 필수 헤더와 하위 호출 전파 기준은 표준 헤더 가이드를 기준으로 합니다.

상황사용 API주의점
현재 거래 ID 조회TransactionContext.currentTransactionId()로그, 감사, 외부 호출 연계 키로 사용합니다.
현재 회원/고객/채널 조회TransactionContext.memberNo(), customerNo(), channelCode()요청 DTO와 헤더 값이 충돌하면 업무 정책을 명확히 둡니다.
업무 처리 중 세부 채널 보정CpfHeaderMutator.put(...)거래 ID, trace/span, Authorization 같은 시스템/민감 헤더는 변경할 수 없습니다.
서비스 간 호출CpfWebClient, CpfRestClientInterceptor표준 헤더는 자동 전파되므로 업무 코드가 반복해서 넣지 않습니다.

실제 조회 EDU 샘플은 XyzQueryEducationController와 XyzQueryEducationService에서 확인합니다. 경로는 /xyz/edu/query/items/{itemId}, /xyz/edu/query/items, /xyz/edu/query/items/page, /xyz/edu/query/items/keyset, /xyz/edu/query/headers입니다.

EDU 조회 샘플 기준

조회 샘플은 단건, 목록, offset 페이징, keyset 페이징, 검색, 정렬, DTO 매핑, MyBatis Mapper, Service 트랜잭션 경계, Controller Swagger 예시, 하위 호출 헤더 자동 전파를 각각 분리합니다. EDU 코드는 학습 목적이므로 한글 주석을 자세히 달고, 운영 코드는 기능 단위 주석으로 유지합니다.
