개발 가이드

신규 업무 개발자가 CPF 기준으로 Controller, Service, Mapper, SQL, 테스트, 문서를 맞추기 위한 가이드입니다.

API Versioning

신규 외부 공개 API는 /api/v1/{module}/{domain}를 기본으로 사용합니다. 기존 샘플 경로인 /mbr, /accounts, /xyz/edu, /adm/api는 교육/호환 경로로 유지합니다.

신규 모듈 생성 기준

산출물기준
패키지cpf.{module}.{domain}.controller/service/mapper/dto
ControllerSwagger tag, summary, description, request/response example 작성
Service상태 변경 트랜잭션 경계와 권한/감사 사유 검증 위치
Mapper동적 문자열 SQL 조립 금지, parameter binding 사용
SQL{module}_{업무명} 테이블과 공통 감사 컬럼, 한글 COMMENT
Testvalidation, service 단위, mapper 또는 DB smoke

복붙 기준은 아래 템플릿과 xyz 교육 샘플을 함께 사용합니다.

구성표준 경로/역할
Controllercpf.{module}.{domain}.controller. 신규 공개 API는 /api/v1/{module}/{domain}를 우선 사용하고, 호환 API는 별도 설명을 남깁니다.
Servicecpf.{module}.{domain}.service. 상태 변경, 권한 검사, 감사 사유 검증, 트랜잭션 경계를 담당합니다.
Mappercpf.{module}.{domain}.mapper. SQL Injection 방지를 위해 문자열 결합 대신 parameter binding을 사용합니다.
DTOcpf.{module}.{domain}.dto. Bean Validation과 Swagger example을 함께 작성합니다.
SQL{module}_{업무명} 테이블명, 공통 감사 컬럼, 한글 COMMENT, FK/UK/index 명명 규칙을 적용합니다.
Testvalidation, service 단위, mapper 또는 smoke 검증 중 변경 위험에 맞는 테스트를 추가합니다.

공통 기준

항목기준
응답/오류pfw_response_code, pfw_message에 등록하고 Swagger example과 테스트를 함께 갱신
Transaction상태 변경 Service method에 @Transactional 적용. Controller와 Mapper에는 두지 않습니다.
검색/정렬정렬 컬럼은 allowlist로 제한하고 페이징 기본값/최대값을 둡니다.
Validationrequest DTO에는 Bean Validation, 복합 규칙은 custom validator 사용
외부 호출timeout, retry, circuit breaker, correlation id 전파를 기본으로 합니다.

거래 헤더 표준

모든 업무 API는 CPF 표준 거래 헤더를 Swagger, TransactionContext, 외부 호출 전파, 거래 로그 적재 기준과 일치시켜야 합니다.

구분헤더기준
필수X-Request-Type, X-Original-Channel-Code, X-Channel-Code요청 유형과 채널 추적을 위해 Controller 진입 시 검증합니다.
추적X-Transaction-Id, X-Trace-Id, X-Span-Id, X-Parent-Span-IdX-Transaction-Id는 필수이며 34자리 공식 규격으로 검증합니다. Trace/Span 값은 없으면 CPF가 생성해 응답/로그/외부 호출에 전파합니다.
클라이언트X-Api-Version, X-Client-App-Id, X-Client-Version, X-Caller-Service, X-Caller-Instance-Id배포 버전, 호출자, 인스턴스 추적에 사용합니다.
업무 추적X-Correlation-Id, X-Idempotency-Key, X-Member-No, X-Customer-No, X-User-Id외부 대조, 중복 요청 방지, 회원/고객 추적에 사용합니다.

Batch 개발 샘플

CPF 배치는 Spring Batch를 표준 엔진으로 사용하고, ADM 배치 관제는 Spring Batch 실행 결과와 CPF 운영 메타 테이블을 함께 조회합니다.

Spring Batch 자체 실행 저장소는 BATCH_* 테이블이며, CPF 운영 화면에서 검색하고 감사하는 데이터는 pfw_batch_* 테이블입니다. 신규 배치 개발자는 Job bean을 등록하고, 운영 노출이 필요한 Job은 pfw_batch_job seed 또는 ADM 배치 등록으로 메타를 맞춥니다.

신규 개발 시 참고 샘플

ADM 알림AdmNotificationController, AdmNotificationService, NotificationSender 구조를 참고합니다. Controller는 요청/응답만 처리하고, SQL/감사/발송은 Service와 Sender로 분리합니다.
ADM 다운로드AdmDownloadController, AdmDownloadService를 참고합니다. 다운로드 유형은 화이트리스트로 제한하고, CSV 생성, 민감정보 마스킹, 감사 로그 저장을 한 서비스에서 처리합니다.
배치 스케줄러PfwBatchScheduleService, PfwBatchLockService, PfwBatchExecutionTargetService, PfwBatchScheduler를 참고합니다. 스케줄 판정, lock, 실행 대상 기록, 실행 요청을 분리합니다.
업무 관리자/대외 샘플bizadm은 ADM 내부 운영 기능이 아니라 업무 프로젝트 관리 화면의 구현 예시입니다. exs는 외부기관 송수신과 거래/전문 로그 분리 예시입니다.
품질 Gate신규 Java/HTML/SQL 변경은 checkJavaFormat, checkHtmlDocs, checkSqlStandard, checkTransactionIdStandard를 통과해야 합니다. 외부 서버가 필요한 OpenAPI smoke는 openApiSmoke로 별도 실행합니다.
