CPF 개발 가이드

이 문서는 신규 업무 기능을 만드는 개발자가 Controller, Service, Repository 또는 Mapper, DTO, SQL, Swagger, EDU, 테스트를 같은 기준으로 작성하도록 안내합니다.

PFW가 제공하는 표준 기능은 업무 모듈에서 직접 재구현하지 않습니다. 거래 ID, 표준 응답, 예외, 배치 실행, OpenAPI 공통 설명, 거래 로그는 PFW API와 annotation을 사용합니다.

순서작업기준참고 파일

Service와 트랜잭션

상황사용 기준주의
단순 조회@Transactional(readOnly = true) 또는 DB 접근 단일 query대량 조회는 paging, limit, 정렬 기준을 명확히 둡니다.
등록/수정/삭제Service public method에 쓰기 트랜잭션을 둡니다.Controller에서 DB 작업을 직접 호출하지 않습니다.
감사 로그주 트랜잭션 실패와 별도 보존이 필요하면 REQUIRES_NEW를 검토합니다.감사 사유 누락 시 저장하지 않습니다.
외부 호출 포함DB lock과 외부 호출을 한 트랜잭션 안에 오래 묶지 않습니다.timeout, retry, circuit breaker, correlation id를 남깁니다.
배치Job 자체는 Spring Batch가 관리하고 실행 요청은 CpfBatchLauncher를 통과합니다.업무 개발자가 JobLauncher를 직접 호출하지 않습니다.

PFW 공통 API 사용법

기능사용 API개발자가 해야 할 일하지 말아야 할 일
거래 IDTransactionContext, TransactionIdGenerator요청 헤더를 검증하고 내부 호출에 전파합니다.업무별 별도 거래 ID 포맷을 만들지 않습니다.
표준 예외CpfBusinessException, CpfValidationException, CpfNotFoundException의미 있는 메시지와 오류 상황을 구분합니다.내부 stack trace를 응답으로 노출하지 않습니다.
WebClientCpfWebClient, CpfHttpClientPropertiestimeout, retry, header 전파를 표준으로 사용합니다.임의 HTTP client를 제각각 만들지 않습니다.
BatchCpfBatchLauncher, CpfBatchExecutionRequestJob ID, 파라미터, 요청자, 사유를 표준 모델로 전달합니다.PFW lock과 운영 이력을 우회하지 않습니다.
Workflow@CpfWorkflow, @CpfWorkflowStep멀티스텝 업무의 실패 정책과 추적 단위를 명확히 남깁니다.단순 로그 문자열만으로 단계 상태를 표현하지 않습니다.
