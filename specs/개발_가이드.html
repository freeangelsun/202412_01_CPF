CPF 개발 가이드

이 문서는 신규 업무 기능을 만들 때 Controller, Service, Repository 또는 Mapper, DTO, SQL, Swagger, EDU, 테스트를 어떤 기준으로 연결해야 하는지 설명합니다. 기능을 완료라고 말하려면 실제 파일 경로와 테스트 증거가 있어야 합니다.

Controller만 만들고 Service, Repository, SQL, 테스트가 없으면 완료가 아닙니다. 문서만 있는 기능, sample 데이터만 있는 기능, in-memory 저장소만 있는 운영 기능도 완료가 아닙니다.

PFW와 CMN 경계

영역책임개발자가 사용하는 방식금지 기준
PFWCPF 프레임워크 코어. 거래 ID, 표준 헤더, 응답/예외, 거래 로그, OpenAPI, WebClient, Batch 공통 API, 프레임워크 메시지/설정/응답코드 제공cpf.pfw 패키지의 API, AOP, Filter, Interceptor, Facade를 호출합니다.업무 모듈이 pfw_* 테이블을 직접 갱신하지 않습니다.
CMNCPF 적용 프로젝트의 업무 공통. 업무 채번, 업무 알림, 업무 로그, 파일/전문/MQ 보조 기능 제공cpf.cmn 패키지의 업무 공통 Service를 호출하고, 필요하면 프로젝트 공통 기능을 확장합니다.PFW 자체 표준 기능을 CMN에서 다시 만들지 않습니다.
업무 모듈BIZADM, MBR, ACC, EXS처럼 실제 도메인 기능 구현Controller-Service-Repository/Mapper-DTO-SQL-Test 흐름을 지킵니다.운영 기능에 하드코딩 목록이나 sample 저장소를 사용하지 않습니다.

신규 기능 개발 절차

순서작업완료 기준참고 파일
1주제영역과 DB prefix 결정테이블명은 {주제영역}_{업무명} lower snake case이고, 공통 컬럼 created_by, created_at, updated_by, updated_at을 포함합니다.specs/sql/40_business_modules_schema.sql
2API 경로와 DTO 설계API version, 요청/응답 DTO, validation, 표준 오류 응답을 정의합니다.MbrController, BizAdmOperationController, ExsFlowController
3Service 트랜잭션 경계 작성쓰기 작업은 Service 계층에서 트랜잭션을 잡고, 감사 사유와 권한 검사를 저장소 접근 전에 수행합니다.ExsOperationService, AdmDownloadService
4Repository 또는 Mapper 작성운영 구현체는 DB 기반으로 동작하며, datasource가 없으면 명확한 오류를 냅니다.BizAdmOperationRepository, ExsOperationRepository, MemberMapper
5SQL, Flyway, all install 동기화분리 SQL, 전체 설치 SQL, smoke SQL, Flyway baseline 또는 변경 migration이 서로 맞습니다.specs/sql, specs/sql/migration/flyway
6Swagger와 EDU 연결주요 API에 tag, summary, description이 있고, 개발자가 참고할 EDU 또는 가이드가 연결됩니다.xyz/src/main/java/cpf/xyz/edu
7테스트 작성정상, 오류, 권한 또는 감사 사유, DB 저장 또는 저장 차단 흐름을 테스트합니다.BizAdmAuthServiceTest, MbrAuthServiceTest, ExsOperationServiceTest, AdmDownloadServiceTest
8문서와 매트릭스 갱신README, 개발 가이드, 관리자 가이드, SQL 가이드, 기능 구현 매트릭스, 안정화 리포트가 실제 코드와 일치합니다.이 HTML 가이드 묶음

Controller 작성 기준

항목기준예시
경로업무 API는 /api/{domain}, 운영 API는 /adm/api/{domain} 형식을 사용합니다./api/exs/inbound, /adm/api/batch/jobs
거래 로그추적 대상 API에는 @CpfTransaction을 붙이고, 거래 ID와 trace를 응답 헤더로 전파합니다.ExsFlowController.receiveInbound
권한ADM write, delete, execute, retry, stop API는 화면 버튼 숨김에 그치지 않고 서버에서 권한을 검사합니다.AdmApiAuthFilter, AdmPermissionService
오류 응답검증 오류, 업무 오류, 시스템 오류를 표준 오류 구조로 반환합니다.CpfGlobalExceptionHandler
Swagger@Tag와 @Operation은 운영자가 이해할 수 있는 한글 설명으로 작성합니다.BIZADM-Operations, EXS-Operations

Service와 트랜잭션 기준

상황기준주의
단순 조회Service에서 조회 조건을 정규화하고 Repository는 DB 결과만 반환합니다.limit와 검색 조건은 반드시 상한을 둡니다.
등록/수정/삭제Service public method에서 트랜잭션을 잡고, validation과 권한검사를 먼저 수행합니다.Controller에서 JdbcTemplate이나 Mapper를 직접 호출하지 않습니다.
감사 대상 작업사유 검증은 저장소 접근보다 먼저 수행합니다.토큰, 비밀번호, 마스킹 해제, 다운로드, 권한 변경은 사유 누락 시 DB 쓰기를 하지 않습니다.
대외 호출timeout, retry, correlation id, 오류 로그 기준을 함께 둡니다.DB lock을 잡은 상태에서 외부 호출을 오래 유지하지 않습니다.
배치업무 Job은 PFW CpfBatchLauncher와 Spring Batch JobRepository 흐름을 따릅니다.업무 개발자가 JobLauncher를 제각각 직접 호출하지 않습니다.

검증 명령

검증명령판정
전체 테스트.\gradlew.bat test --offline테스트 성공. 실패하면 기능 완료로 보고하지 않습니다.
품질 gate.\gradlew.bat qualityGate --offlineUTF-8, SQL, 문서, sample, 보안 seed, 기능 증거 검사 통과
BIZADM 인증.\gradlew.bat :bizadm:test --offline --tests cpf.bizadm.auth.service.BizAdmAuthServiceTest로그인 성공 이력, 실패 횟수 증가, refresh token hash 저장과 원문 미저장 확인
BIZADM 서비스.\gradlew.bat :bizadm:test --offline --tests cpf.bizadm.operation.service.BizAdmOperationServiceTest마스킹과 감사 사유 흐름 확인
MBR 인증.\gradlew.bat :mbr:test --offline --tests cpf.mbr.bse.service.MbrAuthServiceTest로그인 성공/실패 이력, refresh token hash 저장과 폐기 토큰 차단 확인
EXS 서비스.\gradlew.bat :exs:test --offline --tests cpf.exs.operation.service.ExsOperationServiceTesttoken hash, 사유 검증, retry, 수신 로그 확인
ADM 다운로드.\gradlew.bat :adm:test --offline --tests cpf.adm.opr.service.AdmDownloadServiceTest다운로드 정책과 차단 흐름 확인
OpenAPIpowershell -NoProfile -ExecutionPolicy Bypass -File scripts/smoke-openapi.ps1앱 기동 후 JSON 검증. 앱 미기동이면 미검증으로 기록합니다.

완료 판정 체크리스트

기능 완료는 소스가 있는 것만으로 끝나지 않습니다. Controller, Service, Repository 또는 Mapper, DTO, SQL, Swagger, EDU 또는 가이드, 정상/오류/권한/DB 테스트 증거가 함께 있어야 합니다.
