권한 모델

ADM 화면 메뉴 필터링은 adm_role_menu.READ_YN을 기준으로 합니다. 등록, 수정, 삭제, 실행, 재수행, 중지, 잠금해제, 비밀번호 초기화 API는 서버 필터에서 메뉴 권한과 버튼 권한을 다시 검사합니다.

bizadm은 ADM 내부 기능이 아니라 CPF 적용 프로젝트의 업무 관리자 화면 샘플입니다. ADM은 공통 운영, 감사, 권한 기준을 제공하고 업무 도메인별 관리자 기능은 별도 업무 모듈에서 구현합니다.

역할기본 범위
ADM_ADMIN전체 메뉴와 전체 버튼/행위 허용
ADM_DEV_OPERATOR로그, 배치, 캐시, 코드, 메시지, 응답코드, 설정, 동적 로그레벨 운영
ADM_BIZ_OPERATOR업무 운영에 필요한 로그/배치/캐시 일부 실행과 공통 기능 조회
ADM_VIEWER조회 전용

통합 로그 관제

메뉴조회 대상
온라인 거래 로그pfw_transaction_log, pfw_transaction_log_detail
감사 로그adm_audit_log의 before/after/diff, 감사 사유, 대상 정보
배치 관제BATCH_*, pfw_batch_job, pfw_batch_schedule, pfw_batch_job_relation, pfw_batch_execution_target, pfw_batch_execution, pfw_batch_step_execution
운영 작업 로그adm_operation_log, pfw_batch_operation_log
로그 일자 기준LOG_DATE는 거래 시작 시각인 START_TIME의 날짜를 기준으로 저장합니다. 적재 지연이 있어도 트랜잭션 ID 앞 17자리 일시와 운영 조회 기준이 어긋나지 않게 하기 위함입니다.
서버 인스턴스MODULE_ID, WAS_ID, SERVER_INSTANCE_ID, host, process, thread 정보를 함께 조회해 이중화 환경의 처리 위치를 추적합니다.

로그 상세는 요약, 헤더, request, response, error, detail, diff, step log 탭으로 구분합니다. JSON은 pretty print와 복사/다운로드를 제공하고, 고정길이 전문은 필드명, 시작위치, 길이, 값, 설명을 표 형태로 표시합니다.

회원 관리

기능운영 기준
회원 목록/상세회원번호, 고객번호, 로그인 ID, 이름, 이메일, 휴대폰, 상태, 채널, 권한 기준으로 조회하고 상세에서 권한/로그인/거래/감사 이력을 연결합니다.
회원 등록/수정/상태 변경버튼 권한과 API 권한을 모두 검사하고 감사 사유, before/after diff를 남깁니다.
회원 권한관리회원 역할, 서비스 접근 권한, 임시 권한, 권한 만료일, 권한 회수 이력을 mbr_member_role, mbr_member_role_history에 기록합니다.
탈퇴/복구/삭제 정책물리 삭제보다 상태 변경과 감사 추적을 우선합니다. 실제 삭제가 필요하면 보존기간과 법적 근거를 운영 절차에 남깁니다.

비밀번호 관리정책 조회, 초기화, 강제 변경, 잠금 해제, 실패 횟수 초기화, 세션 폐기

배치 운영수동 실행, 실패 재수행, 중지, 스케줄 enable/disable, 수행 시뮬레이션, 관계 조회, 수행 대상 조회를 제공하고 모든 변경 작업은 감사 사유 필수

운영/보안 절차

절차운영 기준
Secret 관리운영자는 ADM에서 secret 원문을 조회하지 않습니다. 암호화 값은 마스킹하고 변경 이력과 감사 사유만 남깁니다.
DB 권한DDL은 migration 계정으로만 수행하고, 앱 계정으로 DDL이 가능하면 보안 이슈로 분류합니다.
민감정보 로그password, token, 주민번호, 계좌번호, 휴대폰 일부 같은 민감정보는 원문 저장과 다운로드를 금지합니다.
오류 대응운영자는 사용자에게 trace id와 표준 응답코드만 안내하고, 상세 stack trace는 내부 로그에서 권한별로 확인합니다.
세션 정리만료 세션은 scheduler 후보로 정리하고, 운영자가 필요 시 세션 조회/폐기 API를 통해 강제 종료합니다.

배치 관제 상세

기능운영 기준
Spring Batch 저장소BATCH_JOB_INSTANCE, BATCH_JOB_EXECUTION, BATCH_STEP_EXECUTION은 Spring Batch가 직접 사용하는 표준 JobRepository 테이블입니다.
CPF 운영 메타pfw_batch_execution.spring_batch_execution_id로 Spring Batch 실행 ID를 연결하고, 운영자 조회/검색/감사 로그는 CPF 메타 테이블에 저장합니다.
영업일만 수행pfw_batch_schedule.business_day_only_yn, calendar_id, holiday_policy로 휴일 수행 여부를 판단합니다.
수행 시뮬레이션ADM의 수행 시뮬레이션 버튼은 기준일과 조회 일수를 받아 영업일 여부, 수행 가능 여부, 사유를 표시합니다.
선행/트리거 관계pfw_batch_job_relation에서 선행 Job, 후행 Job, 트리거 조건, 필수 상태를 조회합니다. 그래프 UI는 이후 시각화 고도화 대상입니다.
수행 대상 인스턴스pfw_batch_execution_target에서 수행 대기/배정/완료 대상 인스턴스, 업무 기준일, 예정 수행 일시를 확인합니다.
알림pfw_notification_rule, pfw_notification_delivery_log는 배치 실패와 보안 이벤트의 운영 알림 기준과 발송 이력을 관리합니다.

알림, 다운로드 감사, 배치 스케줄러 운영

알림 관리ADM 알림 화면과 /adm/api/notifications API에서 알림 규칙 조회, 등록, 수정, 비활성화, 테스트 발송을 수행합니다. 운영 변경은 감사 사유를 필수로 남깁니다.
다운로드 감사거래 로그, 오류 로그, 배치 실행 이력, 알림 발송 이력은 /adm/api/downloads/csv 공통 API로 내려받습니다. 다운로드 사유, 마스킹 여부, 요청자, IP, User-Agent는 adm_download_audit_log에 기록합니다.
배치 자동 실행cpf.batch.scheduler.enabled=true일 때 PfwBatchScheduler가 pfw_batch_schedule 기준으로 실행 대상을 판정합니다. 운영자는 배치 화면의 스케줄러 1회 실행 버튼으로 due schedule 판정을 smoke 검증할 수 있습니다.
권한 기준알림은 NOTIFICATION_*, 다운로드는 DOWNLOAD_*, 스케줄러 수동 실행은 BATCH_SCHEDULER_RUN 버튼 권한을 사용합니다. UI 버튼 노출과 서버 API 권한 검사를 함께 적용합니다.
