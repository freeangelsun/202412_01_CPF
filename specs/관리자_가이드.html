CPF 관리자 가이드

ADM은 CPF 프레임워크를 운영자가 조회, 조치, 감사할 수 있게 하는 운영 콘솔입니다. UI 버튼 표시만으로 보안을 완성하지 않고 서버 권한검사, 감사 사유, before/after diff, 민감정보 마스킹을 함께 적용해야 합니다.

브라우저 실제 클릭 검증을 수행하지 않은 항목은 화면 검증 완료로 보고하지 않습니다. 정적 smoke는 메뉴와 API marker 확인이며, runtime smoke는 앱 기동과 API 응답 검증입니다.

온라인 거래 관제와 배치 관제 분리

구분ADM에서 보는 것현재 상태주의사항
온라인 거래/adm/api/logs로 거래 로그 목록과 상세를 조회하고 /adm/api/transactions로 거래 메타를 조회합니다. JSON pretty와 기본 고정길이 전문 분해 결과를 함께 제공합니다.부분 구현거래 메타 브라우저 UX와 대량 운영 검증은 아직 다음 보강 대상입니다.
복합 거래/adm/api/transaction-groups로 전체 거래 묶음을 검색하고 /segments, /timeline, /headers, /external-logs로 구간별 원인을 분리합니다.부분 구현API와 smoke 스크립트는 추가됐고, 브라우저 timeline UX 검증은 다음 단계입니다.
온라인 오류pfw_transaction_log의 실패 레코드와 상세 로그에서 오류 코드, 메시지, 내부/외부 메시지를 확인합니다.부분 구현별도 pfw_error_log 테이블은 없습니다.
감사/adm/api/audit-logs와 adm_audit_log로 운영 조치 사유, before/after, diff를 조회합니다.부분 구현모든 업무 변경 API까지 감사 강제 범위를 넓히는 작업은 남아 있습니다.
정책 감사/adm/api/log-policy-audits와 pfw_log_policy_audit로 로그 정책 생성, 수정, override, 중지, refresh 이력을 조회합니다.부분 구현일반 감사와 저장소를 분리합니다. 통합 추적 API는 같은 업무 거래 ID 기준으로 정책 감사 후보를 함께 반환합니다.
통합 추적/adm/api/observability/transactions/{transactionGlobalId}, /adm/api/observability/traces/{traceId}, /adm/api/observability/business-transactions/{businessTransactionId}를 사용합니다.부분 구현API wrapper와 runtime smoke 기준은 보강됐고, 브라우저 탭형 상세 UX 클릭 검증은 다음 단계입니다.
배치/adm/api/batch로 Job, schedule, execution, step, worker, relation, target, lock, ghost, calendar를 조회하거나 조치합니다.부분 구현브라우저 클릭 검증과 운영 정책 전체 자동화는 별도 검증이 필요합니다.

배치 운영

항목현재 기준운영자 확인
실행 책임BAT가 PFW Batch 공통 API를 통해 Job을 실행합니다. ADM은 관제와 조치 요청을 담당합니다.실제 실행 workerId와 JobExecutionId를 확인합니다.
영업일 실행스케줄의 영업일 적용 여부, 수행 가능 시간, 휴일 정책을 기준으로 실행 후보를 계산합니다.수행 시뮬레이션 결과와 실제 schedule 상태를 비교합니다.
수행 시뮬레이션/adm/api/batch/schedules/{scheduleId}/simulation 기준으로 수행 예정일을 조회합니다.운영 화면에서는 버튼 권한, 사유 입력, 결과 표시가 함께 필요합니다.
선행/후행/트리거 관계pfw_batch_job_relation 기준으로 관계를 조회합니다.목록과 상세에서 선행 Job, 후행 Job, trigger Job 관계를 확인합니다.
수행 대상pfw_batch_execution_target 기준으로 대기, 배정, 완료 대상을 조회합니다.수행 대기 인스턴스와 실제 execution 관계를 확인합니다.
진행률 관제pfw_batch_execution과 pfw_batch_step_execution의 처리 건수, 진행률, 현재 step, 마지막 heartbeat를 조회합니다.장시간 실행 Job은 heartbeat smoke 기준처럼 주기 갱신 증거를 남겨야 합니다.
JobInstance 파일 로그GET /adm/api/reliability/batch-job-logs와 상세 API로 업무일자, Job명, JobInstance ID별 JSON Lines를 조회합니다.서버는 ${CPF_LOG_ROOT}/{environment}/bat/jobs 아래 정규 파일만 읽고 실제 절대 경로, 심볼릭 링크, 경로 이탈을 허용하지 않습니다.
ghost 조치heartbeat timeout과 RUNNING execution 잔존 여부로 ghost 후보를 판단합니다. 후보 조회 시 자동 감지 이벤트가 pfw_batch_ghost_event에 등록됩니다.FAIL, ABANDON, RELEASE_LOCK 조치는 감사 사유를 필수로 남깁니다.

Reliability 운영 API

PFW 멱등, Broker, 파일전송, 결과 미확정 데이터는 /adm/api/reliability에서 조회합니다. 조회와 수동 작업은 AdmApiAuthFilter의 메뉴·버튼·API 권한을 통과해야 하며 replay와 수동 확정은 감사 사유가 필수입니다.

기능API운영 기준
멱등 처리 조회GET /adm/api/reliability/idempotencyscope, 상태, idempotency key로 검색합니다. 저장 응답 원문은 기본 목록에서 노출하지 않습니다.
Broker 조회GET /adm/api/reliability/broker/outbox, /inbox, /dlqtransactionGlobalId, topic, 처리 상태, worker, 실패 사유를 확인합니다.
DLQ replayPOST /adm/api/reliability/broker/dlq/{messageId}/replayRELIABILITY_REPLAY 권한, 사유, before/after 감사 로그가 필요하며 이미 요청된 건은 거부합니다.
파일전송 이력GET /adm/api/reliability/file-transfersendpoint, 상태, transactionGlobalId, checksum, 크기, 결과 상세를 확인합니다.
BAT JobInstance 로그GET /adm/api/reliability/batch-job-logs, GET /adm/api/reliability/batch-job-logs/{businessDate}/{jobName}/{jobInstanceId}업무일자, Job명, JobInstance ID로 검색하고 최근 JSON Lines를 구조화해 조회합니다. 조회 API는 ADM 서버 권한 필터를 통과해야 합니다.
결과 미확정 조회GET /adm/api/reliability/unknown-results유형, 상태, transactionGlobalId, external key, 다음 조치를 확인합니다.

DB 로그 복구 상태GET /adm/api/reliability/transaction-log-recoverypending, processing, poison, spool 사용량, 누적 복구·실패 건수와 health를 조회합니다.
DB 로그 즉시 복구POST /adm/api/reliability/transaction-log-recovery/runRELIABILITY_RECOVERY_RUN 권한과 감사 사유가 필수이며 before/after 상태를 감사 로그에 남깁니다.

현재 API compile/unit test와 정적 UI JavaScript 검증 범위는 통과했지만 실제 MariaDB 조회와 브라우저 클릭은 미검증입니다. 이 상태를 운영 화면 완료로 해석하지 않습니다.

거래 로그 헤더 분리 조회

ADM 거래 로그 상세는 데이터부와 헤더부를 분리해서 봅니다. 서버 응답은 summary, inboundHeaders, resolvedHeaders, outboundHeaders, responseHeaders, request, response, error, formattedDetails를 반환합니다.

탭의미운영 확인 포인트
수신 헤더클라이언트나 상위 서비스가 보낸 헤더필수 헤더 누락, 채널/회원/고객 식별 값을 확인합니다.
해석 헤더PFW가 보정하거나 생성한 현재 거래 기준 헤더거래 ID, trace/span, correlation, client IP 해석 결과를 확인합니다.
전파 헤더하위 서비스로 전달한 헤더parent transaction/span 관계와 민감정보 미전파 여부를 확인합니다.
응답 헤더클라이언트로 돌려준 헤더응답 거래 ID와 trace ID가 로그와 일치하는지 확인합니다.
