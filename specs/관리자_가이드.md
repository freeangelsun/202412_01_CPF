# FPS 관리자 가이드

## 1. 목적
이 문서는 FPS 프레임워크를 운영하거나 관리하는 사람이 보는 통합 기준입니다.

기존에 흩어져 있던 아키텍처, 배포, DB 설치, 프레임워크 필수 테이블, 파일 로그, 운영자 테이블, 테이블 설계 표준을 이 문서 하나로 통합합니다. 개발자가 코드를 작성할 때 보는 기준은 `개발_가이드.md`, 프로젝트 첫 진입점은 `README.md`입니다.

## 2. 문서 체계

| 문서 | 역할 |
| --- | --- |
| `README.md` | 프로젝트 개요, 빠른 실행, 핵심 링크 |
| `specs/개발_가이드.md` | 개발 표준, 패키지, 거래ID, 트랜잭션, 서비스 호출, 예외, 캐시, OpenAPI |
| `specs/관리자_가이드.md` | 운영/관리 기준, 배포, DB, 로그, 운영 화면, 파일 로그 |

운영자가 보는 문서는 이 파일을 기준으로 합니다. 테이블 설계, 배포, 서비스 호출, 파일 로그처럼 운영 관점이 섞이는 내용은 별도 파일로 늘리지 않고 이 문서에 반영합니다. 개발자가 코딩할 때 직접 지켜야 하는 기준은 `개발_가이드.md`에 반영합니다.

## 3. 모듈 운영 구조

| 모듈 | 성격 | 배포 방식 | 운영 책임 |
| --- | --- | --- | --- |
| `pfw` | 프레임워크 공통 라이브러리 | 업무 JAR에 포함 | 거래ID, 헤더, 로그, 표준 예외, WebClient, 워크플로우 |
| `cmn` | 개발 공통 라이브러리 | 업무 JAR에 포함 | 코드/메시지/설정값 캐시, 공통 유틸, 캐시 리프레시 |
| `mbr` | 회원 업무 서비스 | 독립 JAR/WAS | 회원 API |
| `acc` | 계좌 업무 서비스 | 독립 JAR/WAS | ACC 샘플 업무, MBR 호출 샘플 |
| `xyz` | 교육용 업무 서비스 | 독립 JAR/WAS | 신규 개발자 교육용 API |
| `adm` | 운영 UI 후보 | 별도 JAR/WAS 권장 | 로그 조회, 캐시 리프레시, 코드/메시지/설정 관리, 운영자 관리 |

`pfw`, `cmn`은 포트를 열어 단독 실행하는 서버가 아닙니다. `acc`, `mbr`, `xyz` 같은 실제 서비스가 빌드될 때 라이브러리로 포함됩니다.

## 5. 로컬 실행과 서버 배포

### 5.1 로컬 전체 구동
로컬에서 전체 업무 서비스를 한 번에 띄울 때는 루트 Gradle 태스크를 사용합니다.

```powershell
.\gradlew.bat runLocalServices
```

실행 대상:

| 서비스 | 포트 | 설명 |
| --- | --- | --- |
| `ACC` | `8080` | 계좌 주제영역 |
| `MBR` | `8081` | 회원 주제영역 |
| `XYZ` | `8099` | 교육용 주제영역 |

VS Code나 IntelliJ에서는 Gradle 창에서 `Tasks > application > runLocalServices`를 더블클릭합니다.

### 5.2 로컬 개별 구동

```powershell
.\gradlew.bat :mbr:bootRun
.\gradlew.bat :acc:bootRun
.\gradlew.bat :xyz:bootRun
```

개별 실행 시에도 `pfw`, `cmn`은 자동 포함됩니다.

### 5.3 서버 반영 흐름

```text
소스 반영
  -> Gradle 빌드
  -> 대상 모듈 bootJar 생성
  -> 대상 WAS 또는 서버에 JAR 업로드
  -> 서비스별 환경 변수와 프로파일 확인
  -> 서비스 재기동
  -> 헬스 체크와 거래 로그 확인
```

배포 단위는 반드시 주제영역 서비스 기준입니다.

| 배포 대상 | 빌드 명령 | 산출물 |
| --- | --- | --- |
| MBR 서버 | `.\gradlew.bat :mbr:clean :mbr:bootJar` | `mbr/build/libs/*.jar` |
| ACC 서버 | `.\gradlew.bat :acc:clean :acc:bootJar` | `acc/build/libs/*.jar` |
| XYZ 서버 | `.\gradlew.bat :xyz:clean :xyz:bootJar` | `xyz/build/libs/*.jar` |

전체 검증 빌드:

```powershell
.\gradlew.bat build --offline
```

### 5.4 Jenkins 기준
Jenkins에서는 모듈별 파이프라인 또는 파라미터 기반 공통 파이프라인을 권장합니다.

| 파라미터 | 예시 | 설명 |
| --- | --- | --- |
| `MODULE` | `mbr`, `acc`, `xyz` | 배포할 주제영역 |
| `PROFILE` | `dev`, `test`, `prod` | 실행 프로파일 |
| `TARGET_HOST` | `mbr-was-01` | 배포 대상 서버 |

Windows agent:

```powershell
.\gradlew.bat :%MODULE%:clean :%MODULE%:bootJar --offline
```

Linux agent:

```bash
./gradlew :${MODULE}:clean :${MODULE}:bootJar --offline
```

`MODULE=mbr`이면 MBR 실행 JAR만 만들고, `MODULE=acc`이면 ACC 실행 JAR만 만듭니다. 이때 `pfw`, `cmn`은 별도 배포하지 않습니다.

### 5.5 배포 후 확인

1. 프로세스가 기동되었는지 확인합니다.
2. 포트가 정상 Listen 상태인지 확인합니다.
3. 대표 API 또는 health API를 호출합니다.
4. 응답 헤더에 `X-Transaction-Id`, `X-Trace-Id`가 있는지 확인합니다.
5. `pfwDB.TRAN_LOG`, `pfwDB.TRAN_LOG_DTL`에 거래 로그가 적재되는지 확인합니다.
6. ACC에서 MBR을 호출하는 샘플 거래가 같은 글로벌 거래ID로 이어지는지 확인합니다.
7. 파일 로그에 같은 거래ID가 찍히는지 확인합니다.

## 6. 서비스 주소와 환경 변수
서비스 간 호출 주소는 PFW 공통 설정의 `fps.services.*.base-url` 기준을 사용합니다.

| 키 | 로컬 기본값 | 설명 |
| --- | --- | --- |
| `FPS_SERVICE_ACC_URL` | `http://localhost:8080` | ACC 서비스 주소 |
| `FPS_SERVICE_MBR_URL` | `http://localhost:8081` | MBR 서비스 주소 |
| `FPS_SERVICE_CMN_URL` | `http://localhost:8082` | CMN 서버 확장 예약값 |

현재 `cmn`은 단독 서버로 구동하지 않으므로 `FPS_SERVICE_CMN_URL`은 확장 예약값입니다. 운영에서는 고정 IP보다 DNS, VIP, Kubernetes Service, Config Server 값을 권장합니다.

```text

```

### 7.2 테이블 설계 공통 원칙
테이블 설계 공통 기준은 이 절에서 관리합니다. 이 기준은 업무 테이블, 공통 테이블, 운영 테이블에 모두 적용합니다.

공통 감사 컬럼:

| 컬럼 | 타입 예시 | 기준 |
| --- | --- | --- |
| `CREATED_BY` | `VARCHAR(100)` | 등록자. 기본값은 `SYSTEM` |
| `CREATED_AT` | `DATETIME` | 등록시각. 기본값은 현재시각 |
| `UPDATED_BY` | `VARCHAR(100)` | 수정자. 기본값은 `SYSTEM` |
| `UPDATED_AT` | `DATETIME` | 수정시각. 수정 시 현재시각 |

명명 규칙:

| 대상 | 기준 |
| --- | --- |
| 테이블 | 업무 의미가 드러나는 단수형 또는 프로젝트 표준명 |
| PK | `{table}_id` 또는 업무상 확정된 식별자 |
| 인덱스 | `IDX_{TABLE}_{COLUMN}` |
| 유니크 | `UK_{TABLE}_{COLUMN}` |
| FK | `FK_{FROM}_{TO}` |

인덱스 기준:

- 운영 화면의 검색 조건, 조인 키, 정렬 키를 우선합니다.
- 로그/본문처럼 큰 텍스트 컬럼은 일반 인덱스 대상에서 제외합니다.
- INSERT가 많은 테이블은 인덱스를 과도하게 만들지 않습니다.
- 전문 검색이 필요하면 DB 인덱스보다 로그 분석 시스템 또는 검색 저장소를 검토합니다.

## 8. PFW 필수 로그 테이블
PFW 로그 테이블 기준은 이 절에서 관리합니다. PFW 로그 테이블은 업무 테스트 테이블이 아니라 프레임워크 필수 운영 테이블입니다.

| 테이블 | 용도 |
| --- | --- |
| `pfwDB.TRAN_LOG` | 거래 처리 요약 로그. 목록 화면, 검색, 통계, 감사 기준 |
| `pfwDB.TRAN_LOG_DTL` | 거래 처리 상세 로그. 요청/응답/헤더/워크플로우 상세 키-값 |

### 8.1 거래 추적 기준

| 항목 | 기준 |
| --- | --- |
| 글로벌 거래ID | `TRANSACTION_ID` |
| 컨트롤러 거래ID | `BUSINESS_TRANSACTION_ID` |
| 거래명 | `BUSINESS_TRANSACTION_NAME` |
| 워크플로우 | `WORKFLOW_ID`, `WORKFLOW_INSTANCE_ID`, `WORKFLOW_STEP_ID`, `WORKFLOW_STATUS` |
| 보상 거래 | `COMPENSATION_YN`, `COMPENSATION_TRANSACTION_ID`, `COMPENSATION_TARGET_TRANSACTION_ID`, `COMPENSATION_STATUS` |
| 실행 위치 | `EXECUTION_PACKAGE`, `EXECUTION_CLASS`, `EXECUTION_METHOD`, `EXECUTION_SIGNATURE` |
| 시간 | `START_TIME`, `END_TIME`, `DURATION_MS` |
| 오류 | `ERROR_CODE`, `EXTERNAL_MESSAGE`, `INTERNAL_MESSAGE`, `ERROR_MESSAGE` |

### 8.5 운영 조회 예시
거래ID 기준 전체 흐름:

```sql
SELECT
  LOG_IDX,
  TRANSACTION_ID,
  BUSINESS_TRANSACTION_ID,
  BUSINESS_TRANSACTION_NAME,
  WORKFLOW_INSTANCE_ID,
  WORKFLOW_STEP_ID,
  WORKFLOW_STATUS,
  COMPENSATION_YN,
  COMPENSATION_STATUS,
  EXECUTION_CLASS,
  EXECUTION_METHOD,
  REQUEST_TYPE,
  CHANNEL_CODE,
  MEMBER_NO,
  CUSTOMER_NO,
  MODULE_ID,
  WAS_ID,
  RESPONSE_CODE,
  LOG_TYPE,
  START_TIME,
  END_TIME,
  DURATION_MS
FROM TRAN_LOG
WHERE TRANSACTION_ID = '20260611141234567ACCaccAP010000001'
ORDER BY START_TIME, LOG_IDX;
```

오류 거래 조회:

```sql
SELECT
  LOG_IDX,
  TRANSACTION_ID,
  MODULE_ID,
  WORKFLOW_INSTANCE_ID,
  WORKFLOW_STATUS,
  COMPENSATION_STATUS,
  URI,
  RESPONSE_CODE,
  ERROR_CODE,
  EXTERNAL_MESSAGE,
  ERROR_MESSAGE,
  START_TIME
FROM TRAN_LOG
WHERE LOG_TYPE = 'FAILURE'
ORDER BY START_TIME DESC
LIMIT 100;
```

상세 로그 조회:

```sql
SELECT
  D.DETAIL_KEY,
  D.DETAIL_VALUE,
  D.CREATED_AT
FROM TRAN_LOG L
JOIN TRAN_LOG_DTL D ON D.LOG_IDX = L.LOG_IDX
WHERE L.TRANSACTION_ID = '20260611141234567ACCaccAP010000001'
ORDER BY L.START_TIME, L.LOG_IDX, D.DETAIL_ID;
```

### 8.6 로그 화면 구성 권장
목록 화면 기본 컬럼:

| 컬럼 | 화면 의미 |
| --- | --- |
| `LOG_IDX` | 로그 순번 |
| `TRANSACTION_ID` | 글로벌 거래ID |
| `BUSINESS_TRANSACTION_ID`, `BUSINESS_TRANSACTION_NAME` | 컨트롤러 거래ID/거래명 |
| `WORKFLOW_INSTANCE_ID`, `WORKFLOW_STEP_ID`, `WORKFLOW_STATUS` | 워크플로우 인스턴스/스텝/상태 |
| `COMPENSATION_YN`, `COMPENSATION_STATUS` | 보상 거래 여부/상태 |
| `REQUEST_TYPE`, `ORIGINAL_CHANNEL_CODE`, `CHANNEL_CODE` | 요청 구분/채널 |
| `MEMBER_NO`, `CUSTOMER_NO`, `EXEC_USER`, `CLIENT_IP` | 검색 키와 수행자 |
| `MODULE_ID`, `WAS_ID`, `SEQUENCE_NO` | 처리 주제영역/노드/순번 |
| `HTTP_METHOD`, `URI`, `RESPONSE_CODE`, `LOG_TYPE` | 호출과 결과 |
| `START_TIME`, `END_TIME`, `DURATION_MS` | 시간대별 처리 추적 |

상세 화면은 `TRAN_LOG` 한 건의 요약 정보와 `TRAN_LOG_DTL`의 키-값 목록을 함께 보여줍니다.

## 9. 마스킹과 파일 로그

### 9.1 마스킹 기준
로그에 남기기 전 다음 키는 `***`로 마스킹합니다.

```text
password, passwd, pwd, token, authorization, auth, secret,
ssn, rrn, resident, residentNo, accountNo, accountNumber,
cardNo, cardNumber, pin, otp
```

본문이 너무 길면 일정 길이 이후 `...(truncated)`로 자릅니다.

### 9.2 파일 로그 정책
거래 추적의 기준 데이터는 `pfwDB.TRAN_LOG`, `pfwDB.TRAN_LOG_DTL`입니다. 파일 로그는 장애 분석 시 개발자와 운영자가 빠르게 흐름을 확인하는 보조 수단입니다.

| 항목 | 기준 |
| --- | --- |
| 로그 패턴 | `transactionId`, `traceId`, `spanId`를 MDC에서 출력 |
| 모듈 로그 | 모듈별 일자 파일 |
| PFW/CMN 로그 | 업무 서비스 로그 경로 아래 별도 파일 |
| 거래ID별 로그 | `transactions/yyyy-MM-dd/{transactionId}.{index}.log` |
| 보관 기간 | 30일 |
| 파일 크기 | 거래ID별 파일 10MB 단위 롤링 |
| 보호 장치 | `maxAppenderCount=1000`, `totalSizeCap=2GB` |
| 업무 거래 표시 | MDC `businessTransactionId`를 `BizTranId`로 출력 |
| 동적 진단 표시 | MDC `dynamicLogLevel`을 `DynLevel`로 출력 |

운영 TPS가 높은 시스템에서는 거래ID별 파일이 파일 수를 크게 늘릴 수 있습니다. 운영에서는 DB 거래로그와 로그 수집 시스템을 기준으로 삼고, 거래ID별 파일은 장애 분석용 보조 장치로 제한값을 둡니다.

### 9.3 저장 경로

| 서비스 | 경로 예시 |
| --- | --- |
| ACC | `${LOG_PATH}/acc/transactions/yyyy-MM-dd/{transactionId}.0.log` |
| MBR | `${LOG_DIR}/transactions/yyyy-MM-dd/{transactionId}.0.log` |

ACC에서 MBR을 호출하면 두 WAS에 같은 글로벌 거래ID가 전파됩니다. 같은 거래ID 파일을 ACC와 MBR 각각에서 열면 주제영역별 처리 흐름을 이어 볼 수 있습니다.

## 10. 동적 로그레벨 운영
PFW는 `DynamicTransactionLogLevelService`를 통해 특정 글로벌 거래ID 또는 업무 거래ID만 임시 진단 대상으로 등록할 수 있습니다.

현재 적용 범위:

- PFW 거래 시작/종료/실패 진단 로그를 지정 레벨로 추가 출력합니다.
- `TRAN_LOG_DTL`에 `dynamicLog.rule.id`, `dynamicLog.level`, `dynamicLog.reason`, `dynamicLog.expiresAt`을 남깁니다.
- 파일 로그 MDC에 `DynLevel`이 찍힙니다.

주의:

- 이 기능은 전체 패키지 로거 레벨을 강제로 바꾸는 기능이 아닙니다.
- 운영에서 업무 코드 내부의 모든 DEBUG 로그까지 특정 거래만 허용하려면 Logback TurboFilter 또는 로그 수집 플랫폼 연동을 추가로 설계해야 합니다.
- 운영 화면 구현 시 규칙 저장소는 `admDB.dynamic_log_level_rule`을 사용하고, WAS별 반영은 CMN 캐시 리프레시 이벤트와 같은 방식으로 전파하는 것을 권장합니다.

## 11. 운영자 관리 테이블
운영 UI는 `pfw` 안에 넣지 않고 별도 `adm` 주제영역으로 두는 것을 권장합니다.

| 테이블 | 용도 |
| --- | --- |
| `admDB.operator_user` | 운영자 계정 |
| `admDB.operator_role` | 운영자 역할 |
| `admDB.operator_user_role` | 운영자-역할 매핑 |
| `admDB.operator_audit_log` | 운영자 작업 감사 |
| `admDB.dynamic_log_level_rule` | 특정 거래 동적 로그레벨 규칙 |

보안 기준:

- 운영자 비밀번호는 원문 저장 금지입니다. BCrypt, Argon2 같은 강한 단방향 해시를 사용합니다.
- 운영자 작업은 `operator_audit_log`에 남기고 `transaction_id`, `trace_id`와 연결합니다.
- 로그 상세 화면의 요청/응답 본문은 민감정보 마스킹 결과만 보여줍니다.
- 운영 UI 권한은 조회, 수정, 삭제, 로그 다운로드, 캐시 리프레시, 동적 로그레벨 변경을 분리합니다.

## 12. 공통 코드/메시지/설정 운영
CMN은 코드, 메시지, 설정값 3종을 기본 공통 관리 대상으로 제공합니다.

| 구분 | 테이블 | 운영 기준 |
| --- | --- | --- |
| 코드 | `cmnDB.code_table` | 코드 키, 부모 코드, 사용 여부 기준으로 검색 |
| 메시지 | `cmnDB.message_table` | `EXTERNAL`, `INTERNAL` 메시지를 분리 관리 |
| 설정값 | `cmnDB.config_table` | 설정 타입, 암호화 여부, 사용 여부 관리 |
| 리프레시 이벤트 | `cmnDB.cache_refresh_event` | 다중 WAS 캐시 리프레시 전파 |

CRUD가 성공하면 현재 WAS 캐시는 즉시 갱신되고 `cache_refresh_event`를 통해 다른 WAS도 기본 5초 주기로 변경 이벤트를 감지합니다. 운영에서 더 강한 실시간성이 필요하면 Kafka, Redis Pub/Sub, Spring Cloud Bus 같은 메시징 방식으로 이벤트 조회 부분을 교체할 수 있습니다.

## 13. 트랜잭션 운영 관점
트랜잭션 설계 자체는 `개발_가이드.md`를 따릅니다. 운영자는 장애 조회와 후속 처리를 위해 다음 상태를 구분합니다.

| 상태 | 의미 | 운영 처리 |
| --- | --- | --- |
| `FAILED` | 즉시 실패 확정 | 고객 안내, 원인 분석 |
| `RETRY_PENDING` | 재시도 대기 | 재시도 스케줄과 횟수 확인 |
| `COMPENSATING` | 보상 거래 필요 | 보상 대상 거래와 보상 거래 확인 |
| `PENDING` | 접수/보류 | 후속 배치, 이벤트 처리 확인 |
| `VERIFY_REQUIRED` | 상대 시스템 상태 확인 필요 | 상태조회, 대사 |
| `MANUAL_REQUIRED` | 운영자 판단 필요 | ADM 수동 처리 |
| `IGNORED` | 업무상 무시 가능 | 승인 근거 확인 |

후거래가 실패했다고 무조건 롤백하는 것이 아니라 업무 정책에 따라 접수, 보류, 재시도, 보상, 수동 처리 중 하나로 운영합니다.
