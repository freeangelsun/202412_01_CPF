CPF 배치 개발 가이드

이 문서는 BAT 모듈에서 배치 Job을 개발하고 실행하며, Spring Batch JobRepository와 CPF 운영 메타를 어떻게 연결하는지 설명합니다. README나 프레임워크 구성 가이드의 일부 문단으로 대체하지 않고 별도 기준 문서로 유지합니다.

PFW는 프레임워크 공통 API를 제공하고, BAT는 실행 worker입니다. ADM은 배치 상태를 조회하고 운영 조치를 요청하지만 실제 Job/Step 실행 책임을 갖지 않습니다.

아키텍처 경계

영역책임배치 개발자가 지켜야 할 기준
PFWBatch 공통 launcher, 거래 ID, 표준 상태, 오류, 로그, 감사 기준 제공업무 Job은 PFW 내부 테이블을 직접 갱신하지 않고 PFW Batch API를 사용합니다.
BATSpring Boot 기반 배치 worker, Job/Step 실행, JobRepository 기록Job, Step, Tasklet, Reader, Processor, Writer는 BAT 또는 업무 모듈의 확장 영역에 둡니다.
ADM배치 관제, 수동 실행 요청, 중지, 재수행, ghost 조치, 로그 조회운영 조치는 감사 사유와 권한검사를 거쳐 BAT/운영 메타에 전달합니다.
업무 모듈업무 데이터 처리와 업무별 center-cut adapter업무 DB 변경은 업무 서비스 경계에서 처리하고, PFW DB를 업무 테이블처럼 사용하지 않습니다.

실행 중 heartbeat와 진행률

BAT Job은 CpfBatchRuntimeListener를 Job/Step listener로 등록해 시작과 종료 상태를 CPF 운영 메타에 반영합니다. 장시간 실행 Tasklet 또는 Chunk 처리 루프는 CpfBatchHeartbeatService.recordStepProgress를 주기적으로 호출해 실행 중 처리량을 갱신합니다.

항목저장 위치운영 의미
worker heartbeatpfw_batch_worker.last_heartbeat_atBAT worker process가 살아 있고 어떤 Job/execution을 처리 중인지 판단합니다.
실행 진행률pfw_batch_execution.total_count, processed_count, progress_rate, current_step_nameADM 실행 목록에서 현재 step과 처리 진척을 표시합니다.
Step 진행률pfw_batch_step_execution의 count/progress/heartbeat 컬럼Step 상세에서 처리 건수, 실패 건수, retry/skip, 마지막 heartbeat를 확인합니다.
ghost 자동 감지pfw_batch_ghost_eventheartbeat timeout이 지난 RUNNING execution을 중복 없이 후보 이벤트로 등록합니다.

CPF_BAT_HEARTBEAT_JOB은 heartbeat 검증용 smoke Job입니다. 기본 설정은 cpf.bat.smoke.heartbeat-iterations와 cpf.bat.smoke.heartbeat-sleep-ms로 조정하며, smoke script는 heartbeatCount= step log와 처리 건수 2회 이상 갱신을 확인합니다.

기본 실행 명령

.\gradlew.bat :bat:test --offline
.\gradlew.bat :bat:bootJar --offline
java -jar bat\build\libs\bat-0.0.1-cpf.bat-SNAPSHOT.jar --spring.profiles.active=bat-local
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\smoke-bat-runtime.ps1

jar 파일명은 빌드 설정에 따라 달라질 수 있습니다. 문서의 명령과 실제 산출물 이름이 달라지면 smoke script와 README를 함께 수정합니다.

신규 Job 작성 절차

단계작업검토 기준
1Job 이름, 업무 영역, parameter, 재수행 가능 여부 정의동일 parameter 중복 실행과 재수행 정책이 명확해야 합니다.
2Tasklet 또는 Chunk 방식 선택대량 데이터는 paging/chunk, 단순 운영 작업은 tasklet을 우선 검토합니다.
3실패 정책, skip/retry, checkpoint, idempotency 정의재수행 시 중복 처리나 데이터 손상이 없어야 합니다.
4CPF Batch 운영 메타 연결JobExecutionId, StepExecutionId, transactionGlobalId가 연결되어야 합니다.
5ADM 관제 노출 항목 확인실행 상태, 실패 사유, 처리 건수, step log, 수동 조치 권한을 확인합니다.
6EDU 샘플과 개발 가이드 갱신신규 패턴이면 `xyz` 교육 샘플을 함께 보강합니다.

center-cut 확장 원칙

PFW는 center-cut 대상 목록, item, result 업무 테이블을 직접 소유하지 않습니다. PFW는 표준 인터페이스, 상태, 오류, 로그, 감사, transactionGlobalId 기준을 제공합니다. BAT가 기본 center-cut 구현체를 제공할 수 있고, 업무 영역은 필요 시 업무별 target/item/result 테이블과 adapter를 추가합니다.

확장 지점권장 형태주의사항
대상 조회CenterCutTargetProvider 형태의 업무 adapter업무 DB 소유권을 침범하지 않습니다.
처리 로직CenterCutHandler 형태의 업무 서비스idempotency와 실패 재처리 기준을 문서화합니다.
결과 적재ResultProvider 또는 업무 repository처리 결과와 실패 데이터 보존기간을 정의합니다.
