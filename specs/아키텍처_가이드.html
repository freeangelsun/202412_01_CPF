CPF 아키텍처 가이드

CPF의 가장 중요한 경계는 PFW와 CMN입니다. 이 경계가 흔들리면 프레임워크 기능과 프로젝트별 공통 기능이 섞여 배포, 문서, 교육 샘플, 운영 책임이 불명확해집니다.

PFW는 프레임워크 코어이고, CMN은 프로젝트 공통 영역입니다. 프레임워크가 자체 제공하는 공통 API는 PFW가 제공하고, 프로젝트가 업무 특성에 맞게 확장하는 공통 기능은 CMN에 둡니다.

모듈 책임

모듈책임대표 구현경계 기준
pfw프레임워크 코어표준 헤더, 거래 ID, 표준 응답, 예외, 거래 로그, 로그 정책, OpenAPI 공통 설정, HTTP client 헤더 전파, Batch 공통 APICPF 제품이 기본 제공하는 기능입니다. 업무 프로젝트가 필요하면 PFW API를 호출해 사용합니다.
cmn프로젝트 공통프로젝트별 채번, 업무 알림, 업무 로그, 파일/전문/MQ 보조 기능프로젝트마다 달라지는 공통 업무 기능입니다. 프레임워크 코어로 승격할 기능은 별도 ADR 후 PFW로 이동합니다.
adm운영 콘솔로그, 배치, 권한, 회원, 캐시, 메시지, 코드, 설정, 보안 운영 UI/API운영자가 조회하고 조치하는 기능입니다. 실제 업무 실행은 각 소유 모듈이 담당합니다.
bat배치 workerSpring Batch JobRepository, JobLauncher, Job/Step 실행ADM은 배치 실행 명령과 관제만 담당하고 BAT가 실제 Job을 실행합니다.
xyzEDU 샘플CRUD, 조회, 배치, 외부 호출, 표준 헤더, CMN/PFW 사용 예제교육용 주석은 자세히 작성하고 운영 코드와 분리합니다.

프레임워크 공통 API 제공 원칙

기능PFW 기준CMN 기준
표준 헤더/거래 IDPFW가 자동 검증, 컨텍스트 저장, 하위 호출 전파, 로그 저장, ADM 표시 기준을 제공합니다.CMN은 직접 구현하지 않고 PFW API를 사용합니다.
메시지/코드/응답코드/설정프레임워크 공통 메타와 운영 캐시는 PFW가 소유합니다.프로젝트 전용 코드나 메시지는 CMN 또는 업무 DB에 확장할 수 있습니다.
채번/알림/업무 로그모든 프로젝트에 공통으로 필요한 범용 API는 PFW 승격 후보입니다.프로젝트 정책, 업무키, 보존기간, 채널이 다른 기능은 CMN에서 개발합니다.
운영 관제PFW는 로그/배치/캐시/정책 API와 메타를 제공합니다.프로젝트별 운영 화면 확장은 ADM 또는 업무 관리자 모듈에서 연결합니다.

요청 처리 구조

온라인 요청은 TransactionContextFilter가 거래 ID와 trace/span을 준비하고, TransactionHeaderValidationInterceptor가 필수 헤더와 거래 메타를 검증합니다. Controller는 @CpfTransaction으로 업무 거래 ID를 선언하고, LoggingAspect는 요청/응답/오류/헤더 스냅샷을 거래 로그로 저장합니다.

하위 호출은 CpfWebClientConfig와 CpfRestClientInterceptor를 통해 표준 헤더를 자동 전파합니다. 업무 코드가 직접 헤더 문자열을 조립하지 않도록 CpfHeaderNames와 TransactionContext API를 사용합니다.
