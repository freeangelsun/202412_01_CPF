CPF 운영 매뉴얼

이 문서는 ADM, BAT, 업무 모듈을 운영 환경에서 어떻게 구동, 중지, 점검, 복구할지 정리한 runbook입니다. 코드 기능의 존재와 운영 검증 결과를 혼동하지 않기 위해 실행하지 않은 검증은 항상 미검증으로 기록합니다.

PFW는 library 성격의 프레임워크 코어이므로 독립 process로 구동하지 않습니다. 실제 process로 구동되는 대표 모듈은 ADM, BAT, 업무 애플리케이션입니다.

process별 책임

process구동 책임정상 확인장애 시 우선 조치
ADM운영 콘솔, 권한, 회원, 로그, 배치 관제, 캐시/메시지/코드/설정 관리/adm/api/health, OpenAPI, ADM runtime smoke로그 확인, DB 연결 확인, API 권한/세션 상태 확인
BATSpring Batch Job/Step 실행, JobRepository 기록, CPF 배치 메타 적재/bat/api/health, BAT runtime smoke, workerId 확인RUNNING execution, lock, worker heartbeat, ghost 후보 확인
업무 앱온라인 업무 API와 업무 DB 처리업무 health, 거래 로그, trace ID거래 ID 기준으로 PFW 로그와 업무 로그를 함께 추적
PFW독립 process 없음각 애플리케이션 내부에서 Bean, filter, AOP, facade 동작 확인공통 모듈 버전, 설정, 의존성 충돌 확인

정상 중지 절차

단계조치확인 기준
1신규 Job 실행을 차단합니다.scheduler disabled 또는 운영 조치 로그 확인
2RUNNING Job과 Step을 조회합니다.Spring Batch execution과 CPF execution 상태가 일치해야 합니다.
3가능하면 graceful shutdown을 수행합니다.worker heartbeat 종료, lock 해제, running execution 없음
4종료 후 health 미응답을 확인합니다.process 종료, port 해제, log flush 확인

강제 종료와 ghost 조치

상황판단 기준운영 조치
worker 비정상 종료health 미응답, heartbeat timeout, RUNNING execution 잔존ADM ghost 후보 조회 시 자동 감지 이벤트를 등록하고, 조치 사유를 입력해 FAIL, ABANDON, RELEASE_LOCK 중 선택합니다.
lock 잔존동일 job/parameter 실행이 lock 때문에 차단실행 중 worker 존재 여부를 확인한 뒤 lock 해제 또는 재수행 여부를 결정합니다.
부분 처리 실패step 실패, 처리 건수 불일치, 실패 데이터 존재skip/retry 정책과 checkpoint를 확인한 뒤 재수행 가능 상태인지 판단합니다.

온라인/배치 공통 추적 기준

항목온라인배치운영 확인

traceId/spanId표준 헤더와 응답 헤더로 전파배치 실행 요청을 만든 온라인/운영 거래에서 이어질 수 있음외부 연계 또는 배치 수동 실행 추적 시 함께 확인합니다.
DB 로그pfw_transaction_log, pfw_transaction_log_detailBATCH_*, pfw_batch_*, pfw_batch_operation_log온라인 거래 로그와 배치 메타를 혼동하지 않습니다.
감사운영 조치와 주요 변경은 adm_audit_log실행, 중지, 재수행, ghost, lock 해제는 사유를 남김사유 없는 운영 조치는 완료로 보지 않습니다.
